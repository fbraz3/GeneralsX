## 2026-03-03 (SESSION 70): Fix macOS Build Failure ÔÇö Vulkan SDK Download (FINAL)

**Objective**: Fix GitHub Actions `build-macos` workflow failure: `hdiutil: attach failed - image not recognized`.

**Root Cause Analysis**:
- Original URL `https://sdk.lunarg.com/download/latest/mac/vulkan-sdk.dmg` was returning redirects or HTTP errors (~1426 bytes HTML).
- LunarG's SDK delivery changed: latest version (1.4.341.1) is now distributed as `.zip` containing a macOS app installer bundle.
- Correct direct URL: `https://sdk.lunarg.com/sdk/download/1.4.341.1/mac/vulkansdk-macos-1.4.341.1.zip` (318 MB).
- SHA256 hash for validation: `632cbe96c8ed6ed00c6ce25e3a7738c466134f76586e1c51f1419410d7f9042e`.

**Architecture of Vulkan SDK 1.4.341.1**:
- ZIP contains: `vulkansdk-macOS-1.4.341.1.app` (macOS application bundle)
- Inside `.app`: `/Contents/MacOS/InstallVulkan` (executable installer script)
- Installer accepts command-line parameters for unattended deployment
- Target install path: `$HOME/VulkanSDK/1.4.341.1/` (version-specific directory)

**Fixes Applied**:

1. **`.github/workflows/build-macos.yml`** ÔÇö Updated Vulkan SDK installation strategy (REVERSED PRIORITY):
   - **Strategy 1 (PREFERRED)**: Direct download of versioned SDK (1.4.341.1) with SHA256 validation
     - Download from: `https://sdk.lunarg.com/sdk/download/1.4.341.1/mac/vulkansdk-macos-1.4.341.1.zip`
     - Validate SHA256: `632cbe96c8ed6ed00c6ce25e3a7738c466134f76586e1c51f1419410d7f9042e`
     - Extract ZIP to find `.app` bundle
     - Execute installer: `./vulkansdk-macOS-1.4.341.1.app/Contents/MacOS/InstallVulkan`
     - Install to: `$HOME/VulkanSDK/1.4.341.1/`
     - Retry up to 3 times with 15s delays
   - **Strategy 2 (FALLBACK)**: Homebrew (`brew install --cask vulkan-sdk`)
     - Only attempted if versioned download fails
     - May install older version than 1.4.341.1, but guarantees working installation
     - Better than nothing if LunarG CDN is unreachable
   - **Why reversed**: Versioned download is more reliable (direct URL, SHA256 validated, specific version known to work); Homebrew is fallback in case LunarG CDN is down

2. **Installation parameters**:
   - `--root $HOME/VulkanSDK/1.4.341.1` ÔÇö version-specific path (supports coexistence)
   - `--accept-licenses` ÔÇö non-interactive (skip dialog)
   - `--default-answer` ÔÇö use presets for all prompts
   - `--confirm-command install` ÔÇö execute install (not just preview)

3. **`cmake/dx8.cmake`** ÔÇö Updated Vulkan SDK detection paths:
   - First: Check `$VULKAN_SDK` environment variable
   - Second: Search `$HOME/VulkanSDK/*/` directories (find latest version)
   - Third: Try Homebrew locations (Intel: `/usr/local/Caskroom/vulkan-sdk`, ARM64: `/opt/homebrew/Caskroom/vulkan-sdk`)
   - Validate MoltenVK: `lib/libMoltenVK.dylib` must exist
   - Pass detected path to Meson via `VULKAN_SDK` env var

4. **Diagnostics & Error Handling**:
   - SHA256 mismatch detected early (prevent .app corruption)
   - File size checks reveal HTML error responses (< 10MB)
   - First 20 lines of invalid files printed for debugging
   - Installer failures reported with exit status
   - Graceful fallback if primary strategy fails
   - Warns if MoltenVK not found (system Vulkan may still work)

**Testing in CI**:
1. GitHub Actions will retry 3 times on download failure (15s between attempts)
2. SHA256 validation ensures downloaded file integrity
3. Logs show which strategy succeeded (Homebrew vs manual installer)
4. MoltenVK verification confirms complete installation

**Lessons Learned**:
- LunarG changed distribution format (DMGÔåÆZIP with app installer) ÔÇö needed URL and extraction method updates.
- macOS app installers need explicit `--accept-licenses --confirm-command` for headless operation.
- Version-specific paths ($HOME/VulkanSDK/1.4.341.1/) better than generic ($HOME/VulkanSDK/) for multi-version coexistence.
- SHA256 validation critical for supply-chain security (detect corrupted/HTML responses immediately).

## 2026-03-02 - Session 58: win64-modern LNK2019 Linker Errors Resolved ÔÇö generalszh.exe Links

**TL;DR**: Fixed all remaining linker errors (LNK2019, LNK1181, CMake config errors) blocking `generalszh.exe` on the win64-modern stack. Executable now links: 10.4 MB at `build/win64-modern/GeneralsMD/generalszh.exe`.

### Background

Previous session had applied 7 source fixes (SDL3Mouse, SDL3Keyboard, SDL3Main, WWVegas
CMakeLists, Core WW3D2 CMakeLists). This session continued from that point.

### Issues Found & Fixed

**Issue 1 ÔÇö GENERATOR EXPRESSION: FramGrab.cpp INTERFACE propagation (absolute path)**

`Core/WW3D2/CMakeLists.txt` used `$<$<BOOL:${WIN32}>:FramGrab.cpp>` (relative path) in
an INTERFACE library's `target_sources`. CMake 4.x cannot resolve generator expression
paths to absolute, so the path propagated to GeneralsMD/Generals WW3D2 consumers. Fixed
by using `${CMAKE_CURRENT_SOURCE_DIR}/FramGrab.cpp`.

**Issue 2 ÔÇö z_wwdownload STATIC empty (vcpkg rejects empty static libs)**

WWDownload directory's `CMakeLists.txt` created `z_wwdownload STATIC` with no sources
(only linking `corei_wwdownload PRIVATE`). vcpkg's `_add_library` override on Windows
rejects empty STATIC libraries. Initial attempt using INTERFACE chaining caused
`corei_wwdownload.lib` to appear in the link command (INTERFACE libs have no .lib). Final
fix: list Core WWDownload `.cpp` files explicitly in `target_sources` using absolute
`${CMAKE_SOURCE_DIR}/...` paths. NOTE: `PROJECT_SOURCE_DIR` in nested CMakeLists points
to the subproject root (GeneralsMD), NOT the workspace root ÔÇö use `CMAKE_SOURCE_DIR`.

**Issue 3 ÔÇö FramGrab.cpp: VFW stub body needed for non-ENABLE_FRAME_GRABBER builds**

`framgrab.h` provides a minimal `AVISTREAMINFO` stub when `ENABLE_FRAME_GRABBER` is not
defined. But `FramGrab.cpp` unconditionally used the full VFW API (AVIFileInit, etc.).
Fixed by guarding the full VFW implementation with `#if defined(ENABLE_FRAME_GRABBER) &&
!defined(DISABLE_VFW)` and providing minimal stub constructors/destructors in the `#else`
branch. The stub satisfies the linker; actual AVI recording is disabled until Phase 3.

**Issue 4 ÔÇö SDL3Mouse.cpp C4189 (unused variable treated as error)**

`frame_offset` declared but never used in the Windows code path. Removed the dead variable.

**Issue 5 ÔÇö WWDownload POSIX functions missing on MSVC**

`FTP.cpp` and `Download.cpp` use POSIX functions not in MSVC CRT:
- `strlcpy` / `strlcat` ÔÇö not in Windows SDK; polyfills added in new `wwdownload_compat.h`
- `socklen_t` ÔÇö not in winsock.h (v1); switch to `winsock2.h` + `ws2tcpip.h` (defines `socklen_t`)
- `ARRAY_SIZE` ÔÇö not defined in these TUs; added to `wwdownload_compat.h`
- `_S_IWRITE` / `_S_IREAD` ÔÇö only in Linux path; added `<sys/stat.h>` in Windows block
- `time()` ÔÇö missing `<time.h>` include in `FTP.cpp` Windows block; added
- `timeGetTime()` ÔÇö `<mmsystem.h>` with `WIN32_LEAN_AND_MEAN` doesn't guarantee the function; must `#include <timeapi.h>` directly

**Issue 6 ÔÇö ftp.h / FTP.cpp used winsock.h (Winsock v1) ÔÇö project uses ws2_32.lib**

Changed `ftp.h` to include `<winsock2.h>` instead of `<winsock.h>`. Winsock2 must be
included BEFORE `windows.h` to avoid the inclusion-order conflict. `ftp.h` is included
before the platform block in both `FTP.cpp` and `Download.cpp`, ensuring correct ordering.

## 2026-03-01 - Session 57: Phase 6 win64-modern Audit + Three Bug Fixes

**TL;DR**: Audited all Phase 6 (win64-modern) changes for band-aids and cross-platform regressions. Found and fixed 3 real bugs. No gameplay or Linux build logic was broken.

### Audit Scope

Reviewed all uncommitted Phase 6 diffs: compat headers (`gdi_compat.h`, `string_compat.h`,
`types_compat.h`, `file_compat.h`, `windows_compat.h`, `windows.h`, `windows_wrapper.h`),
CMakeLists changes (`z_gameengine`, `CompatLib`), build scripts, `StackDump.cpp`, `File.h`.

### Bugs Found & Fixed

**Bug 1 ÔÇö CRITICAL: `gdi_compat.h` content duplicated outside `#ifndef _WIN32` guard**

The Phase 6 patch wrapped the GDI stubs with `#ifndef _WIN32` / `#endif` but forgot to
remove the original un-guarded copy below. The file had 438 lines with lines 229ÔÇô438 being
an exact duplicate of lines 8ÔÇô228, but outside any guard. Fixed by truncating to line 225
(ending at `#endif // !_WIN32`). 4 lessons documented in `docs/WORKDIR/lessons/2026-03-LESSONS.md`.

**Bug 2 ÔÇö PERFORMANCE: `target_precompile_headers(z_gameengine ÔÇª)` silently removed**

The call was deleted from `GameEngine/CMakeLists.txt` without explanation. `PreRTS.h` is
safe to use as PCH on all platforms (already has `#ifdef _WIN32` guards for SDK vs compat
shim). Restored. Expected compile-time improvement: 2ÔÇô4├ù for `z_gameengine` full rebuilds.

**Bug 3 ÔÇö LOGIC: `windows_compat.h` used CMake option as C++ preprocessor guard**

`#if !defined(_WIN32) || (defined(_WIN32) && !defined(SAGE_USE_DX8))` ÔÇö `SAGE_USE_DX8` is
a CMake `option()`, never a `-D` define. The condition was always `true` (guard always applied).
Additionally, `windows_compat.h` is Linux-only (shim gate prevents it reaching Windows at all).
Fixed: replaced with unconditional defines and a clear comment.

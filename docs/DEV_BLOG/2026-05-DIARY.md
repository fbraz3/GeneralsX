## 2026-05-26: Replace widl path heuristics with explicit oaidl probes

Latest Windows run `26481114740` still failed on both targets during widl IDL
generation, with the runner warning that no Wine include directory was found.

	`oaidl.idl` instead of relying on `find_path` heuristics

## 2026-05-25: Fix Windows zlib vcpkg wrapper failure after chainload

After run `26421113480`, both Windows matrix jobs failed at
`build/windows64-deploy/vcpkg_installed/x64-mingw-dynamic/share/zlib/vcpkg-cmake-wrapper.cmake`
with `Broken installation of vcpkg port zlib`.

What was done:
- updated `cmake/toolchains/mingw-w64-x86_64.cmake` to switch
	`CMAKE_FIND_ROOT_PATH_MODE_LIBRARY` and `CMAKE_FIND_ROOT_PATH_MODE_INCLUDE`

	broken even when installed

Impact:

	chainload setup

	`Core/Libraries/Source/Compression/CMakeLists.txt`

## 2026-05-11: Fix FPS uncapped and frame counter instability (Issue #132)

Fixed three critical timing bugs affecting both Generals and Zero Hour that caused FPS to be uncapped or frame counter to appear erratic on non-Windows platforms.

What was done:
- **FramePacer API guard**: identified `FramePacer.cpp` using Windows-only `timeBeginPeriod()` / `timeEndPeriod()` API without `#ifdef _WIN32` guards, causing undefined behavior on macOS/Linux. Added `#ifdef _WIN32` guards in constructor/destructor.
- **FPS limit validation**: identified `FrameRateLimit::wait()` missing validation for `maxFps == 0` or unreasonable values, allowing uncapped mode unintentionally. Added early-exit handling for invalid FPS targets.
- **FPS init logic fix** (Generals base root cause): found `GameEngine.cpp` using `if (m_useFpsLimit && m_framesPerSecondLimit <= 0)` which prevented setting default FPS when `m_useFpsLimit=FALSE`. This was why Generals base appeared uncapped while Zero Hour worked—Generals' INI loading left m_framesPerSecondLimit=0 with m_useFpsLimit=FALSE.
- ensured shared `Core/GameEngine/` fixes automatically apply to both Generals base and Zero Hour expansions.

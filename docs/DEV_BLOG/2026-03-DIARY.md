## 2026-03-03 (SESSION 73): Fix macOS Build Failure — CMake Parse Error in dx8.cmake

**Status**: Fixed CMake syntax error in ExternalProject_Add configure command.

**Problem Found**:

CMake config failed with:
```
CMake Error at cmake/dx8.cmake:119:
  Parse error.  Expected "(", got newline with text "".
```

**Root Cause**:

In `ExternalProject_Add(dxvk_macos_build)` `CONFIGURE_COMMAND`, the variable `${VULKAN_SDK_ENV_VAR}` was on its own line:

```cmake
CONFIGURE_COMMAND
  ${CMAKE_COMMAND} -E env
    CC=clang CXX=clang++
    "CFLAGS=..."
    "CXXFLAGS=..."
    "LDFLAGS=..."
    ${VULKAN_SDK_ENV_VAR}    ← On own line!
  ${MESON_EXECUTABLE} setup ...
```

When this variable expands to an empty string, CMake's parser sees:
```
${CMAKE_COMMAND} -E env ... 
                        ← Empty line (just spaces)
${MESON_EXECUTABLE} setup ...
```

This confuses CMake's argument parser, which expects the next token after `-E env` arguments, not a newline.

**Solution Applied**:

Moved `${VULKAN_SDK_ENV_VAR}` to the same line as the `env` command:

```cmake
CONFIGURE_COMMAND
  ${CMAKE_COMMAND} -E env CC=clang CXX=clang++ "CFLAGS=..." "CXXFLAGS=..." "LDFLAGS=..." ${VULKAN_SDK_ENV_VAR}
  ${MESON_EXECUTABLE} setup ${DXVK_BUILD_DIR} ${DXVK_SOURCE_DIR}
    ...
```

When `VULKAN_SDK_ENV_VAR` is empty, the line becomes:
```
${CMAKE_COMMAND} -E env CC=clang CXX=clang++ "CFLAGS=..." "CXXFLAGS=..." "LDFLAGS=..."
```
This is valid and CMake continues to the next line without parse errors.

**Expected Outcome**:
- CMake parses dx8.cmake successfully
- DXVK Meson build configuration can proceed

## 2026-03-02 - Session 58: win64-modern LNK2019 Linker Errors Resolved ÔÇö generalszh.exe Links

**TL;DR**: Fixed all remaining linker errors (LNK2019, LNK1181, CMake config errors) blocking `generalszh.exe` on the win64-modern stack. Executable now links: 10.4 MB at `build/win64-modern/GeneralsMD/generalszh.exe`.

### Background

Previous session had applied 7 source fixes (SDL3Mouse, SDL3Keyboard, SDL3Main, WWVegas
CMakeLists, Core WW3D2 CMakeLists). This session continued from that point.

### Issues Found & Fixed

**Issue 1 ÔÇö GENERATOR EXPRESSION: FramGrab.cpp INTERFACE propagation (absolute path)**

`Core/WW3D2/CMakeLists.txt` used `$<$<BOOL:${WIN32}>:FramGrab.cpp>` (relative path) in
an INTERFACE library's `target_sources`. CMake 4.x cannot resolve generator expression
paths to absolute, so the path propagated to GeneralsMD/Generals WW3D2 consumers. Fixed
by using `${CMAKE_CURRENT_SOURCE_DIR}/FramGrab.cpp`.

**Issue 2 ÔÇö z_wwdownload STATIC empty (vcpkg rejects empty static libs)**

WWDownload directory's `CMakeLists.txt` created `z_wwdownload STATIC` with no sources
(only linking `corei_wwdownload PRIVATE`). vcpkg's `_add_library` override on Windows
rejects empty STATIC libraries. Initial attempt using INTERFACE chaining caused
`corei_wwdownload.lib` to appear in the link command (INTERFACE libs have no .lib). Final
fix: list Core WWDownload `.cpp` files explicitly in `target_sources` using absolute
`${CMAKE_SOURCE_DIR}/...` paths. NOTE: `PROJECT_SOURCE_DIR` in nested CMakeLists points
to the subproject root (GeneralsMD), NOT the workspace root ÔÇö use `CMAKE_SOURCE_DIR`.

**Issue 3 ÔÇö FramGrab.cpp: VFW stub body needed for non-ENABLE_FRAME_GRABBER builds**

`framgrab.h` provides a minimal `AVISTREAMINFO` stub when `ENABLE_FRAME_GRABBER` is not
defined. But `FramGrab.cpp` unconditionally used the full VFW API (AVIFileInit, etc.).
Fixed by guarding the full VFW implementation with `#if defined(ENABLE_FRAME_GRABBER) &&
!defined(DISABLE_VFW)` and providing minimal stub constructors/destructors in the `#else`
branch. The stub satisfies the linker; actual AVI recording is disabled until Phase 3.

**Issue 4 ÔÇö SDL3Mouse.cpp C4189 (unused variable treated as error)**

`frame_offset` declared but never used in the Windows code path. Removed the dead variable.

**Issue 5 ÔÇö WWDownload POSIX functions missing on MSVC**

`FTP.cpp` and `Download.cpp` use POSIX functions not in MSVC CRT:
- `strlcpy` / `strlcat` ÔÇö not in Windows SDK; polyfills added in new `wwdownload_compat.h`
- `socklen_t` ÔÇö not in winsock.h (v1); switch to `winsock2.h` + `ws2tcpip.h` (defines `socklen_t`)
- `ARRAY_SIZE` ÔÇö not defined in these TUs; added to `wwdownload_compat.h`
- `_S_IWRITE` / `_S_IREAD` ÔÇö only in Linux path; added `<sys/stat.h>` in Windows block
- `time()` ÔÇö missing `<time.h>` include in `FTP.cpp` Windows block; added
- `timeGetTime()` ÔÇö `<mmsystem.h>` with `WIN32_LEAN_AND_MEAN` doesn't guarantee the function; must `#include <timeapi.h>` directly

**Issue 6 ÔÇö ftp.h / FTP.cpp used winsock.h (Winsock v1) ÔÇö project uses ws2_32.lib**

Changed `ftp.h` to include `<winsock2.h>` instead of `<winsock.h>`. Winsock2 must be
included BEFORE `windows.h` to avoid the inclusion-order conflict. `ftp.h` is included
before the platform block in both `FTP.cpp` and `Download.cpp`, ensuring correct ordering.
